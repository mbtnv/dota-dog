# AGENTS.md

Правила для разработчиков и coding agents, которые работают в репозитории `dota-dog`.

## Главный принцип

- Не считать CI местом, где "узнаем, сломалось ли". CI должен подтверждать локально проверенный результат.
- Перед `push` локальная ветка должна проходить те же проверки, что и `.github/workflows/ci-cd.yml`.
- Пуш в `main` особенно чувствителен: после успешного CI запускается production deploy. `main` должен оставаться в состоянии "можно выкатывать".

## Среда разработки

- Использовать Python `3.13`.
- Управление зависимостями и запуск команд выполнять через `uv`.
- После клона репозитория и после изменения зависимостей запускать:

```bash
uv sync --frozen --dev
```

- Если меняется `pyproject.toml`, вместе с ним должен быть обновлен и закоммичен `uv.lock`.

## Обязательные проверки перед push

Запускать из корня репозитория:

```bash
uv run ruff format .
uv run ruff check .
uv run ty check
uv run pytest
```

Дополнительно полезно сверить форматирование именно в CI-режиме:

```bash
uv run ruff format --check .
```

Правило простое:

- если упал `ruff`, `ty` или `pytest`, сначала исправляем проблему, потом коммитим и пушим;
- не оставляем "починю в следующем коммите после CI";
- не пушим код, который локально не был прогнан через обязательные проверки.

## Минимальный рабочий цикл

1. Синхронизировать окружение: `uv sync --frozen --dev`.
2. Внести изменения маленькими, понятными шагами.
3. Во время работы гонять точечные проверки на изменяемую часть.
4. Перед `push` обязательно прогнать полный набор: `ruff format`, `ruff check`, `ty check`, `pytest`.
5. Только после этого пушить ветку или открывать PR.

## Правила по типам изменений

- Изменили Python-код в `src/` или `tests/`: перед `push` обязателен полный `uv run pytest`, даже если локально гонялись только точечные тесты.
- Изменили модели, SQLAlchemy-схему, репозитории или поведение базы данных: проверить, нужна ли миграция в `src/dota_dog/infra/db/migrations/versions/`.
- Изменили миграции или код, завязанный на схему: локально проверить применение миграций через `uv run alembic upgrade head`.
- Изменили зависимости: держать в sync `pyproject.toml` и `uv.lock`.
- Изменили команды запуска, переменные окружения, процесс деплоя или onboarding: обновить `README.md`, `docs/` и примеры env-файлов, если это требуется.

## Что не должно попадать в коммит

- секреты, токены, реальные `.env` файлы;
- временные `print`, отладочные костыли и закомментированный код "на потом";
- несвязанные рефакторинги в одном коммите с функциональным изменением;
- изменения, которые не проходят локальные проверки.

## Качество изменений

- Следовать текущей конфигурации `ruff` из `pyproject.toml`.
- Не игнорировать type errors без явной причины; код должен проходить `uv run ty check`.
- Предпочитать небольшие, сфокусированные изменения вместо широких переписываний без необходимости.
- Если меняется пользовательское поведение, рядом должны быть тесты или обновление существующих тестов.

## Рекомендация для тех, кто часто забывает проверки

- Имеет смысл поставить локальный `pre-push` hook, который запускает:

```bash
uv run ruff format --check .
uv run ruff check .
uv run ty check
uv run pytest
```

- Если хук слишком тяжелый для каждой попытки пуша, минимумом все равно остается ручной прогон этих команд перед отправкой изменений в репозиторий.

## Короткий чеклист перед push

- окружение синхронизировано через `uv sync --frozen --dev`, если это было нужно;
- `uv run ruff format .` выполнен;
- `uv run ruff check .` проходит;
- `uv run ty check` проходит;
- `uv run pytest` проходит;
- миграции, документация и lockfile обновлены, если изменение их затрагивает;
- в коммите нет секретов, дебажного мусора и случайных файлов.
