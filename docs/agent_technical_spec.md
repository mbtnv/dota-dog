# Техническое задание для агента: Dota Dog v2

## 1. Контекст

В репозитории есть legacy-реализация в [old_code/old_code.py](/Users/maxim/Documents/dev/repos/dota-dog/old_code/old_code.py), которая:

- опрашивает OpenDota по списку игроков;
- хранит состояние в `players.json`;
- отправляет в Telegram topic уведомления о новых матчах;
- использует захардкоженные пути, токен, `chat_id`, `thread_id` и таймзону;
- не поддерживает команды бота, БД, миграции, тесты и нормальную эксплуатацию.

Новая версия должна воспроизвести текущий функционал и расширить его до production-ready сервиса.

## 2. Что делает legacy-скрипт

Текущее поведение, которое нужно сохранить:

- для каждого игрока из [old_code/players.json](/Users/maxim/Documents/dev/repos/dota-dog/old_code/players.json) опрашивается OpenDota `recentMatches`;
- новые матчи определяются сравнением `match_id > last_match_id`;
- по каждому новому матчу формируется HTML-сообщение с:
  - результатом `Win/Lose`;
  - героем;
  - KDA;
  - временем окончания и длительностью матча;
  - HD/TD/HH;
  - GPM/XPM;
  - last hits;
  - game mode / lobby type / party size;
  - ссылкой на Dotabuff match;
- перед серией матчей игрока добавляется блок с именем и winrate;
- сообщение отправляется в конкретный Telegram chat topic;
- после успешного прохода обновляется `last_match_id` для игрока.

Ключевые ограничения legacy:

- состояние хранится в JSON, а не в БД;
- только один чат и один thread;
- нет защиты от дублей на уровне БД;
- нет истории матчей для отчетов;
- нет управления через Telegram;
- нет миграций, контейнеризации, тестов и CI-friendly tooling;
- таймзона захардкожена в коде.

## 3. Цель новой версии

Нужно реализовать Telegram-бота и фоновые задачи на Python, которые:

- публикуют новые матчи отслеживаемых игроков Dota 2;
- хранят историю матчей и конфигурацию в БД;
- умеют строить отчеты за день, неделю и месяц;
- позволяют добавлять и удалять игроков через команды бота в групповом чате;
- готовы к локальному запуску и деплою через Docker.

## 4. Обязательный функционал

### 4.1. Мониторинг матчей

- Бот должен периодически опрашивать OpenDota.
- Базовый интервал опроса: `15` минут, настраивается через env.
- Источник для realtime-уведомлений: endpoint, эквивалентный legacy `recentMatches`.
- Для каждого игрока нужно сохранять `last_seen_match_id` и историю матчей в БД.
- Уведомления должны быть идемпотентными: повторная обработка одного и того же матча не должна приводить к повторной отправке сообщения.
- При временной ошибке OpenDota должны быть retry с backoff и нормальный лог ошибки.
- Нужно учесть, что для day/week/month отчетов истории из одного `recentMatches` недостаточно: сервис обязан сохранять каждый новый матч в БД и поддерживать механизм backfill/resync после простоя.

### 4.2. Формат уведомления о новом матче

Нужно сохранить смысл текущего сообщения, но оформить код так, чтобы шаблон был вынесен в отдельный formatter/service.

Минимальное содержимое уведомления:

- имя игрока и ссылка на профиль;
- win/lose;
- герой;
- KDA;
- время окончания матча;
- длительность;
- GPM/XPM;
- HD/TD/HH;
- last hits;
- режим игры;
- тип лобби;
- размер пати;
- ссылка на Dotabuff матч.

### 4.3. Отчеты за период

Необходимо поддержать отчеты за:

- день;
- неделю;
- месяц.

Отчет должен быть доступен в двух режимах:

- автоматически по завершению периода;
- по команде бота.

Требования к отчетам:

- границы периода считаются в таймзоне чата или thread, а не захардкожены в коде;
- автоотчеты должны отправляться в тот же чат/topic, где ведется отслеживание;
- ручной отчет должен уметь строиться по всем игрокам topic либо по одному конкретному игроку;
- данные для отчета берутся из БД, а не считаются только по последнему ответу OpenDota.

Минимальное содержимое отчета:

- количество матчей;
- wins / losses;
- winrate;
- суммарные и средние kills / deaths / assists;
- средний GPM/XPM;
- топ героев периода;
- лучший и худший стрик в периоде;
- ссылка или идентификатор игрока.

Дополнительно допустимо включить:

- среднюю длительность матчей;
- средний hero damage;
- распределение по solo/party;
- разбивку по ranked / turbo / normal.

### 4.4. Управление игроками через бота в групповом чате

Требование пользователя: добавление нового ID должно происходить через бота в чате, а не в личной переписке с ботом.

Обязательные правила:

- команды управления игроками работают только в group / supergroup / forum topic;
- в личном чате бот должен отвечать, что операция недоступна и ее нужно выполнять в группе;
- по умолчанию добавлять/удалять игроков могут только админы чата или список разрешенных пользователей;
- игрок должен привязываться к конкретному `chat_id` и `message_thread_id`;
- один и тот же игрок может быть отслеживаемым в нескольких topic независимо.
- интерфейс должен быть командным, без обязательного чтения произвольных сообщений, чтобы бот корректно работал в группах с Telegram privacy mode.

Минимальный набор команд:

- `/track <account_id> [alias]` - добавить игрока в текущий чат/topic;
- `/untrack <account_id|alias>` - убрать игрока из текущего chat/topic;
- `/players` - показать список отслеживаемых игроков в текущем chat/topic;
- `/report <day|week|month> [account_id|alias]` - построить отчет;
- `/status` - показать состояние сервиса для текущего topic: количество игроков, время последнего опроса, время следующего опроса.

Желательное поведение:

- бот должен принимать `account_id`, ссылку Dotabuff или OpenDota profile URL;
- перед добавлением игрока нужно валидировать, что OpenDota возвращает корректный профиль;
- если игрок уже отслеживается в текущем topic, бот должен вернуть понятное сообщение без дубля записи;
- после добавления игрока бот должен отправить подтверждение с отображаемым именем и идентификатором.

### 6.2. Python best practices

- весь новый код должен быть typed;
- использовать `async` стек там, где это оправдано: bot, HTTP, scheduler, DB;
- не хранить токены и chat ids в коде;
- конфигурация только через env и `.env.example`;
- разделить domain logic, infra, handlers, jobs, repositories;
- не писать SQL и Telegram formatting вперемешку с бизнес-логикой;
- все внешние интеграции должны быть за интерфейсами / адаптерами;
- использовать structured logging;
- подготовить миграции БД с первого коммита, а не откладывать на потом;
- покрыть тестами критичные сценарии: дедупликация, отчеты, команды `/track` и `/report`.

## 7. Архитектура

### 7.1. Общая схема

Сервис состоит из двух логических частей:

- Telegram bot process, который обрабатывает команды и отправляет сообщения;
- scheduler/background jobs, которые опрашивают OpenDota и отправляют автоотчеты.

Допустимы два варианта:

- один процесс, где бот и scheduler живут в одном event loop;
- два отдельных контейнера от одного кода: `bot` и `worker`.

Предпочтительный вариант для production: отдельные контейнеры `bot` и `worker`, чтобы polling Telegram и фоновые задачи не мешали друг другу.

- состояние опроса (`last_seen_match_id`, `last_poll_at`, ошибки);
- информацию об отправленных отчетах;

## 8. SQLite vs Postgres

Нужно поддержать оба варианта на уровне приложения через SQLAlchemy, но выбрать явный production default.

### 8.1. SQLite

Плюсы:

- минимальный порог входа;
- удобно для локальной разработки и простого single-container деплоя;
- не нужен отдельный DB container;
- достаточно для маленького проекта с одним worker.

Минусы:

- хуже переносит конкурентные записи и несколько процессов;
- есть риск блокировок при scheduler + bot + отчетах;
- миграции и изменение схемы менее удобны;
- сложнее масштабировать и резервировать;
- хуже подходит, если бот будет работать в нескольких чатах и часто писать историю матчей.

### 8.2. Postgres

Плюсы:

- лучше для конкурентной записи и нескольких контейнеров;
- надежнее для production;
- удобнее индексы, аналитические запросы и миграции;
- проще делать backup/restore;
- легче расширять проект без смены БД.

Минусы:

- выше операционная сложность;
- нужен отдельный контейнер/сервис;
- локальная разработка чуть тяжелее.

### 8.3. Рекомендация

- Для локальной разработки и быстрого MVP можно использовать SQLite.
- Для production и для архитектуры с отдельными `bot`/`worker` контейнерами рекомендованный default: Postgres.
- В коде нужно поддержать оба DSN:
  - `sqlite+aiosqlite:///...`
  - `postgresql+asyncpg://...`

Если нужно выбрать одно решение сразу без компромиссов, лучше брать Postgres и не тратить потом время на миграцию с SQLite.

## 9. Docker и docker-compose

Обязательно подготовить:

- `Dockerfile` c multi-stage build;
- `docker-compose.yml` для локального запуска;
- `.env.example`;
- named volume для БД или data directory;
- healthcheck для приложения и, если используется Postgres, для БД.

Минимальный compose-сценарий:

- `bot` - Telegram bot handlers;
- `worker` - polling OpenDota + period reports;
- `db` - Postgres, если выбран production-like локальный режим.

Для SQLite допускается упрощенный dev profile без `db` сервиса, но production compose должен быть рассчитан на Postgres.

Задача считается выполненной, если:

- бот запускается через `uv` локально и через Docker;
